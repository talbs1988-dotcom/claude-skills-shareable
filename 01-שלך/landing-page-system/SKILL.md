# סיסטם דף נחיתה, מהרעיון ועד דף חי

הסיסטם הזה קיים כי דף נחיתה "כמעט גמור" הוא דף מבוזבז: בלי קוד מעקב הקמפיין לא נמדד, בלי חיבור למערכת דיוור הלידים נעלמים, בלי דף תודה הלקוח לא יודע מה הצעד הבא. לכן עוברים את השלבים בסדר, בלי לדלג.

חשוב: בעל העסק שמולך הוא לא איש טכני. תסביר כל מונח במילים פשוטות, תשאל שאלה אחת בכל פעם, ותעשה בעצמך את כל מה שטכני. הוא מביא את התשובות, אתה מביא את הביצוע.

## שלב 0 — קודם כול לקרוא, ורק אז לשאול

לפני שמתחילים, לחפש ולקרוא את שני הקבצים האלה:

1. **`CLAUDE.md`** בתיקיית העבודה של הלקוח — שם נמצאים הסיפור של העסק, הקול, קהל היעד והביוגרפיה (אם הלקוח עבר את אשף המותג בסדנה). כל קופי שנכתוב חייב להישען על מה שכתוב שם
2. **`brand-config.md`** בתיקיית העבודה — הפרטים הטכניים של הלקוח: פיקסל, דומיין, מערכת דיוור, צבעים, דפים משפטיים

- אם `brand-config.md` לא קיים — להריץ את אשף ההגדרה: קרא את [references/setup-wizard.md](references/setup-wizard.md)
- אם קיים אבל חסר בו פרט שצריך עכשיו — לשאול רק על מה שחסר ולעדכן את הקובץ
- **אסור לנחש** פיקסל, דומיין או קישור משפטי. אין נתון בקובץ? שואלים את הלקוח

## כללי ברזל

1. **CTA אחיד** — טקסט כפתור אחד לכל הדף, ספציפי למוצר, חוזר בכל המיקומים
2. **תאימות חוקית בכל דף** — נגישות, בנר קוקיז, קישורים משפטיים, checkbox תקנון. בלי זה לא עולים לאוויר
3. **אסור להגיד "הדף מוכן" או למסור קישור חי** בלי להציג ללקוח את טבלת 8 הבדיקות (שלב 4)
4. **קופי מהמילים של הלקוח** — מהסיפור שלו, מהמלצות אמיתיות שלו. לא ממציאים ציטוטים, לא ממציאים תוצאות
5. **תכנון לפני בנייה** — קודם מאשרים עם הלקוח את המבנה והקופי, רק אז כותבים קוד

## שלב 1 — קופי

קרא את [references/copy-guide.md](references/copy-guide.md): מבנה 11 הסקציות של דף מכירה, כללי כתיבה, ואיך משתמשים בסיפור מה-CLAUDE.md של הלקוח.

עבודה סקשן-סקשן: קודם מבנה כללי לאישור, אחר כך כל סקציה בנפרד לאישור, ורק בסוף מחברים. לא מציפים את הלקוח במסמך ענק.

## שלב 2 — בנייה

קרא את [references/build-guide.md](references/build-guide.md): הסטאק, המיתוג מתוך brand-config, חוויית משתמש נכונה, והתאימות החוקית המלאה.

## שלב 3 — בדיקות איכות לפני עלייה

- הפרויקט נבנה נקי (אם Next.js: `npx tsc --noEmit` + `npm run build` בלי שגיאות)
- שינוי טקסט גורף = לחפש בכל קבצי הקומפוננטות שלא נשארה גרסה ישנה באף מקום
- סריקת קופי: ה-CTA זהה בכל המיקומים, אין טעויות כתיב, הלשון אחידה (זכר/נקבה כפי שהוחלט)
- דחיסת תמונות: תמונה ראשית עד 200KB, תמונות פנים עד 100KB, תמונות קטנות עד 50KB

## שלב 4 — צ'קליסט 8 הבדיקות לפני עלייה לאוויר

קרא את [references/deploy-checklist.md](references/deploy-checklist.md) ועבור על כל 8 הסעיפים: קוד מעקב, נגישות, טופס מחובר, כפתורים, תמונות, שמירת קוד, פריסה, דף תודה.

**הפרוטוקול:** לסמן ✅ / ❌ / ❓ לכל סעיף ולהציג ללקוח טבלת סטטוס. כל ❓ = לעצור ולשאול. גם אם הלקוח אומר "תעלה כבר", עוברים קודם על הצ'קליסט יחד. רק כשכל ה-8 ✅ הדף עולה.

## שלב 5 — עלייה לאוויר

- פריסה דרך Vercel (חינמי): אם יש ללקוח GitHub מחברים אוטומטית, אם לא, פורסים ישירות
- דומיין: לפי מה שהוגדר ב-brand-config (דומיין שלו / תת-דומיין / כתובת Vercel זמנית)
- מפתחות וסודות (פיקסל, API של מערכת הדיוור) נשמרים במשתני סביבה, לעולם לא בתוך הקוד
- בדיקת אמת על הקישור החי: למלא את הטופס באמת ולעקוב אחרי כל הנתיב עד הסוף

## שלב 6 — אחרי שהדף באוויר

- לעדכן ב-`brand-config.md` את כתובת הדף החי ותאריך העלייה
- לוודא שהליד מהבדיקה הגיע ליעד (מערכת הדיוור / הטבלה / המייל) תוך דקה
- אם הדף ישמש קמפיין ממומן: לוודא שקוד המעקב יורה על הדף החי
